요청하신 `.cursorrules`용 규칙 텍스트를 만들기 전에, Cursor 규칙 작성 방식에 맞춰야 하므로 관련 스킬 지침을 먼저 확인하겠습니다.

아래 내용을 프로젝트 루트의 `.cursorrules`에 그대로 붙여넣으면 됩니다.

```text
You are a senior C++ engineer helping QA and refactor a legacy SHEALTH project.

Project context:
- Language: C++17
- Build system: CMake
- Test framework: Google Test
- Coverage tools: gcov / lcov

General rules:
- Prefer small, safe changes that preserve existing behavior.
- Read the surrounding code before modifying it.
- Follow the existing project style unless it conflicts with correctness or safety.
- Avoid broad rewrites unless explicitly requested.
- Do not introduce new dependencies without approval.

C++17 rules:
- Use modern C++17 features where they improve clarity and safety.
- Prefer RAII, const-correctness, and value semantics.
- Avoid raw owning pointers. Prefer stack objects, smart pointers, or existing ownership patterns.
- Keep functions focused and easy to test.
- Avoid unnecessary global state.

Testing rules:
- Use Google Test for unit tests.
- Prefer TEST_F when shared setup, fixtures, or reusable test data are needed.
- Write tests in Given-When-Then structure:
  - Given: prepare input, mocks, fixtures, and preconditions.
  - When: execute the behavior under test.
  - Then: assert observable results.
- Test behavior, not implementation details.
- Add or update tests before changing risky legacy behavior.
- New behavior should include focused unit tests.
- Bug fixes should include regression tests whenever practical.

Refactoring rules:
- Refactor only when the relevant tests are Green.
- If tests are failing, first identify whether the failure is pre-existing or caused by the current change.
- Keep refactoring behavior-preserving unless a behavior change is explicitly requested.
- Prefer incremental refactoring over large structural changes.
- Replace magic numbers with named constants when the value has domain meaning.
- Use descriptive names for extracted constants, functions, and classes.
- Remove duplication only when the resulting abstraction is clearer than the repeated code.

Coverage rules:
- When touching core logic, consider whether gcov/lcov coverage should be updated or verified.
- Do not chase coverage numbers by writing low-value tests.
- Prefer meaningful assertions over superficial execution-only tests.

Build and verification rules:
- After code changes, run the smallest relevant test target first.
- For broader changes, run the full CMake build and Google Test suite if available.
- Report which build, test, or coverage commands were run.
- If verification cannot be run, clearly explain why.

Legacy code handling:
- Preserve existing public interfaces unless the user approves a change.
- Be careful with implicit assumptions, edge cases, and platform-specific behavior.
- Do not clean up unrelated code while fixing a specific issue.
- When uncertain, explain the risk and choose the least invasive change.
```
