# SHEALTH 프로젝트 Cursor AI 규칙 보고서

## 1. 작성 목적

본 문서는 SHEALTH C++17 프로젝트에서 Cursor AI가 코드 수정, 테스트 작성, QA, 리팩토링 작업을 수행할 때 일관되게 따라야 할 기준을 정리한 보고서입니다.

프로젝트 루트의 `.cursorrules` 파일에 정의된 규칙을 기준으로 하며, 레거시 코드의 안정성을 유지하면서 점진적으로 품질을 개선하는 것을 목표로 합니다.

## 2. 적용 대상

- 프로젝트: SHEALTH
- 언어 표준: C++17
- 빌드 시스템: CMake
- 테스트 프레임워크: Google Test
- 커버리지 도구: gcov / lcov
- 주요 작업 범위:
  - 레거시 코드 QA
  - 단위 테스트 작성 및 보강
  - 회귀 테스트 추가
  - 동작 보존 리팩토링
  - 빌드 및 테스트 검증

## 3. 기본 작업 원칙

Cursor AI는 SHEALTH 프로젝트에서 시니어 C++ 엔지니어의 관점으로 작업해야 합니다.

기존 동작을 보존하는 작고 안전한 변경을 우선하며, 코드를 수정하기 전에는 주변 구현과 호출 흐름을 먼저 확인해야 합니다. 프로젝트의 기존 스타일을 따르되, 명확성이나 안전성을 해치는 부분이 있다면 개선 방향을 사용자에게 설명해야 합니다.

명시적인 요청 없이 대규모 재작성이나 구조 변경을 수행하지 않으며, 새로운 외부 의존성은 사용자 승인 없이 추가하지 않습니다.

## 4. C++17 코드 작성 기준

C++17 기능은 코드의 명확성과 안전성을 높이는 경우에만 사용합니다.

RAII, const-correctness, 값 기반 설계를 우선하며, 소유권이 있는 raw pointer 사용은 피해야 합니다. 필요한 경우 스택 객체, 스마트 포인터, 또는 기존 코드베이스의 소유권 패턴을 따릅니다.

함수는 작고 테스트하기 쉽게 유지하고, 불필요한 전역 상태를 만들지 않습니다.

## 5. 테스트 작성 기준

단위 테스트는 Google Test를 사용합니다.

공통 준비 코드, fixture, 재사용 테스트 데이터가 필요한 경우에는 `TEST_F`를 우선 사용합니다. 테스트는 Given-When-Then 구조를 따릅니다.

- Given: 입력값, fixture, mock, 사전 조건을 준비합니다.
- When: 테스트 대상 동작을 실행합니다.
- Then: 관찰 가능한 결과를 검증합니다.

테스트는 구현 세부사항보다 외부에서 관찰 가능한 동작을 검증해야 합니다.

위험한 레거시 동작을 변경할 때는 가능한 한 먼저 테스트를 추가하거나 기존 테스트를 갱신합니다. 새로운 기능에는 집중된 단위 테스트를 추가하고, 버그 수정에는 가능한 한 회귀 테스트를 포함합니다.

## 6. 리팩토링 기준

리팩토링은 관련 테스트가 Green 상태일 때만 진행합니다.

테스트가 실패 중인 경우에는 먼저 해당 실패가 기존 실패인지, 현재 변경으로 인해 발생한 실패인지 확인해야 합니다. 명시적인 요구가 없는 한 리팩토링은 동작 보존을 원칙으로 합니다.

큰 구조 변경보다 작은 단계의 점진적 리팩토링을 선호합니다. 도메인 의미가 있는 매직 넘버는 이름 있는 상수로 치환하는 것을 권장하며, 추출한 상수, 함수, 클래스에는 의도를 드러내는 이름을 사용합니다.

중복 제거는 새 추상화가 반복 코드보다 더 명확할 때만 수행합니다.

## 7. 커버리지 관리 기준

핵심 로직을 수정할 때는 gcov/lcov 커버리지 확인 또는 갱신 필요성을 검토합니다.

단순히 커버리지 수치를 올리기 위한 낮은 가치의 테스트는 작성하지 않습니다. 실행만 하는 테스트보다 의미 있는 assertion을 포함한 테스트를 우선합니다.

## 8. 빌드 및 검증 기준

코드 변경 후에는 가장 작은 관련 테스트 타깃부터 실행합니다.

영향 범위가 넓은 변경은 가능하면 전체 CMake 빌드와 Google Test suite를 실행합니다. 작업 완료 시 실행한 빌드, 테스트, 커버리지 명령을 사용자에게 보고해야 합니다.

검증을 실행할 수 없는 경우에는 그 이유를 명확히 설명합니다.

## 9. 레거시 코드 취급 기준

사용자 승인 없이 기존 public interface를 변경하지 않습니다.

암묵적 전제, 경계 조건, 플랫폼 의존 동작에 주의해야 하며, 특정 문제를 해결하면서 무관한 코드를 함께 정리하지 않습니다.

판단이 불확실한 경우에는 위험을 설명하고 가장 침습도가 낮은 변경을 선택합니다.

## 10. 기대 효과

본 규칙을 적용하면 Cursor AI가 SHEALTH 프로젝트에서 다음과 같은 방향으로 일관되게 동작할 수 있습니다.

- 레거시 코드 변경 시 기존 동작 보존
- Google Test 기반 테스트 품질 향상
- Given-When-Then 구조를 통한 테스트 가독성 확보
- Green 상태 기반의 안전한 리팩토링 수행
- 매직 넘버 상수화를 통한 코드 의도 명확화
- gcov/lcov 기반 커버리지 검토 문화 유지
- CMake 기반 빌드 및 검증 절차의 일관성 확보

## 11. 결론

`.cursorrules` 파일에 정의된 규칙은 SHEALTH 프로젝트의 레거시 C++ 코드 품질을 안정적으로 개선하기 위한 Cursor AI 작업 지침입니다.

Cursor AI는 모든 코드 변경에서 테스트 Green 상태, 동작 보존, 점진적 리팩토링, 명확한 검증 보고를 우선해야 합니다.
