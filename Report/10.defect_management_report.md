# SHEALTH BMI 결함 관리 문서 작성 보고서

## 1. 작성 목적

본 문서는 SHEALTH BMI 프로그램의 결함 관리 체계를 정립하기 위해 `docs/defect_report.md`를 작성한 결과를 정리한 보고서입니다.

요구사항은 QA 리드 엔지니어 관점에서 결함 분류 체계, 결함 보고서 템플릿, 품질 메트릭 수집 계획, GitHub Issues 연동 워크플로우를 Markdown 문서로 정리하고 `docs` 폴더에 저장하는 것이었습니다.

## 2. 작업 대상

- 결함 관리 문서: `docs/defect_report.md`
- 기존 결함 목록: `docs/defect_list.md`
- 요구사항 분석 문서: `docs/requirements_analysis.md`
- 테스트 계획서: `docs/test_plan.md`

기존 문서에서 사용 중인 `DEF-###` 결함 ID 형식, `Severity`, `ItemType`, P0/P1/P2 테스트 우선순위, C++17/Google Test/CMake 기반 테스트 환경을 기준으로 결함 관리 문서를 구성했습니다.

## 3. 결함 분류 체계 작성 내용

결함 분류는 `Severity`와 `ItemType`의 2축 체계로 정의했습니다.

`Severity`는 다음 4단계로 구분했습니다.

| Severity | 기준 |
|---|---|
| Critical | 프로그램 중단, 데이터 손상, 메모리 오염, CI 전면 실패 등 릴리스 차단 수준 |
| Major | 핵심 요구사항 위반 또는 주요 계산 결과 오류 |
| Minor | 제한된 조건에서 발생하며 우회 가능한 결함 |
| Info | 결함 후보, 개선 제안, 문서/절차 보완 항목 |

`ItemType`은 다음 5종으로 구분했습니다.

| ItemType | 기준 |
|---|---|
| Functional | BMI 계산, 분류, 비율 산출 등 기능 요구사항 불일치 |
| DataValidation | CSV 파싱, 누락 컬럼, 비숫자 값, 음수/0 입력 등 데이터 검증 문제 |
| TestFailure | Google Test, CTest, Golden Master 등 자동화 테스트 실패 |
| Tooling | 빌드, 커버리지, CI, 실행 환경, 명령어 호환성 문제 |
| Documentation | 요구사항, 테스트 계획, 결함 목록, README 간 불일치 또는 누락 |

또한 `Severity x ItemType` 매트릭스를 작성해 각 조합별 판단 기준을 명확히 했습니다. 예를 들어 `Major x DataValidation`은 컬럼 누락, 비숫자 값, 분모 0 등으로 핵심 결과 오류를 유발하는 결함으로 정의했고, `Info x Tooling`은 도구 개선 제안이나 자동화 후보로 정의했습니다.

## 4. 결함 보고서 템플릿

신규 결함 등록 시 사용할 표준 템플릿을 작성했습니다.

템플릿은 다음 항목을 포함합니다.

- 기본 메타데이터: `Severity`, `ItemType`, `Status`, `Priority`, 발견 단계, 관련 요구사항/테스트, 담당자, 발견일, 수정 버전/커밋
- 재현 절차: 환경, 입력 데이터, 실행 명령
- 기대 결과: 요구사항 또는 테스트 계획 기준 정상 동작
- 실제 결과: 실패 로그, 반환값, 출력, 커버리지 리포트 등 관찰 결과
- 원인 분석: 코드 위치, 데이터 조건, 분기 누락, 테스트 기대값 오류
- 수정 내용: 수정 요약, 영향 범위, 변경 테스트, 문서 갱신 여부
- 검증 결과: 재실행 테스트, 통과/실패 결과, 커버리지 변화, 남은 리스크

이 템플릿은 기존 `docs/defect_list.md`의 간단한 표 기반 결함 추적을 보완하며, 재현과 검증 근거가 필요한 결함을 상세 관리할 수 있도록 설계했습니다.

## 5. 품질 메트릭 수집 계획

품질 메트릭은 테스트 결과, 커버리지, 결함 발견 단계, 결함 처리 효율을 함께 볼 수 있도록 구성했습니다.

| 메트릭 | 산출 방식 | 목표/판단 기준 |
|---|---|---|
| 테스트 통과율 | `passed / total * 100` | P0/P1 100%, 전체 테스트 100% Green |
| 라인 커버리지 | 커버리지 도구 리포트 | 전체 90% 이상, `SHealth.cpp` 95% 이상 |
| 분기 커버리지 | 커버리지 도구 리포트 | 핵심 계산/검증 로직 85% 이상 |
| 단계별 결함 발견율 | `단계별 결함 수 / 전체 결함 수 * 100` | 요구사항/테스트 설계 단계 발견 비율 향상 |
| 결함 재오픈율 | `reopened / resolved * 100` | 5% 이하 |
| 평균 해결 시간 | Open부터 Resolved까지 걸린 시간 | Critical 1영업일, Major 3영업일 이내 우선 관리 |

단계별 결함 발견율은 요구사항 분석, 테스트 설계, 단위 테스트, 통합 테스트, 회귀 테스트, CI, 운영/사용자 제보 단계로 나누어 집계하도록 정의했습니다. 이를 통해 결함이 늦은 단계에서 발견되는 비율을 줄이고, 테스트 설계와 요구사항 분석 단계의 예방 효과를 확인할 수 있습니다.

## 6. 커버리지 도구 기준

현재 프로젝트는 C++17 기반이므로 `gcov/lcov`를 중심으로 커버리지 수집 절차를 작성했습니다.

권장 흐름은 별도 커버리지 빌드 디렉터리에서 Debug 빌드를 수행하고, `ctest` 실행 후 `lcov`로 커버리지 데이터를 수집한 뒤 외부 의존성과 테스트 코드를 제외해 HTML 리포트를 생성하는 방식입니다.

```bash
cmake -S . -B build-coverage -DCMAKE_BUILD_TYPE=Debug \
  -DENABLE_COVERAGE=ON \
  -DCMAKE_CXX_FLAGS="--coverage -O0 -g" \
  -DCMAKE_EXE_LINKER_FLAGS="--coverage"

cmake --build build-coverage
ctest --test-dir build-coverage --output-on-failure

lcov --capture --directory build-coverage --output-file build-coverage/coverage.info
lcov --remove build-coverage/coverage.info \
  "*/_deps/*" "*/src/test/*" "/usr/*" \
  --output-file build-coverage/coverage.filtered.info
genhtml build-coverage/coverage.filtered.info --output-directory build-coverage/coverage-report
```

추후 Java 또는 Python 모듈이 추가될 가능성을 고려해 다음 도구 기준도 함께 포함했습니다.

| 언어 | 도구 | 대표 명령 |
|---|---|---|
| C++ | `gcov/lcov` | `lcov --capture`, `genhtml` |
| Java | `JaCoCo` | `mvn clean test jacoco:report`, `./gradlew test jacocoTestReport` |
| Python | `pytest-cov` | `python -m pytest --cov=. --cov-report=term-missing --cov-report=html` |

## 7. GitHub Issues 연동 워크플로우

선택 항목으로 GitHub Issues 연동 워크플로우를 작성했습니다.

주요 흐름은 다음과 같습니다.

1. 결함 발견 시 `[DEF-###] <요약>` 형식으로 Issue를 생성합니다.
2. Issue 본문은 결함 보고서 템플릿을 사용합니다.
3. Severity, ItemType, Status, Priority를 라벨로 관리합니다.
4. 테스트 실패 결함은 실패 테스트명, 실행 명령, 최소 입력 데이터를 첨부합니다.
5. 수정 PR은 `Fixes #이슈번호` 형식으로 Issue와 연결합니다.
6. CI Green, 리뷰 승인, 검증 결과 작성 후 Issue를 닫습니다.

권장 상태 전이는 다음과 같이 정의했습니다.

```text
Open -> Triage -> In Progress -> Resolved -> Verified -> Closed
                    \-> Blocked
                    \-> Deferred
                    \-> Rejected
```

## 8. 기존 QA 산출물과의 연결성

이번 결함 관리 문서는 기존 산출물과 다음 방식으로 연결됩니다.

| 기존 산출물 | 연결 방식 |
|---|---|
| `docs/requirements_analysis.md` | 결함의 기대 결과와 요구사항 위반 여부 판단 기준 |
| `docs/test_plan.md` | P0/P1/P2 테스트 우선순위, 커버리지 목표, 테스트 완료 기준 |
| `docs/defect_list.md` | 현재 등록된 `DEF-###` 결함 목록과 상태 추적 |
| `src/test/cpp/SHealthBMITest.cpp` | 단위 테스트 실패 및 회귀 검증 근거 |
| `src/test/cpp/TexttestFixture.cpp` | Golden Master 기반 콘솔 출력 회귀 검증 근거 |

특히 요구사항 분석 문서에서 도출된 주요 리스크인 BMI 25.0 미분류, 분모 0, 컬럼 누락/비숫자 입력 예외, `height == 0`, 10,000건 초과 배열 접근은 결함 분류 체계에서 `Major` 또는 `Critical` 후보로 관리할 수 있습니다.

## 11. 후속 권장 사항

1. `docs/defect_list.md`의 기존 결함 항목에 발견 단계, 상태, 우선순위 필드를 추가해 `docs/defect_report.md` 템플릿과 맞춥니다.
2. GitHub Issues를 사용할 경우 권장 라벨을 실제 저장소에 생성하고, 결함 등록 시 템플릿을 적용합니다.
3. CI에서 `ctest` 결과와 커버리지 리포트를 자동 수집해 테스트 통과율과 커버리지 메트릭을 지속적으로 갱신합니다.
4. 릴리스 단위로 단계별 결함 발견율을 집계해 결함이 늦은 단계에서 발견되는 비율을 추적합니다.
