## 3. 수행한 검증

최종 보고서의 정량 수치를 확인하기 위해 기존 MSVC 빌드와 별도 GCC coverage 빌드를 사용했습니다.

### 3.1 CTest 검증

다음 명령으로 기존 테스트 스위트를 실행했습니다.

```powershell
ctest --test-dir build --output-on-failure
```

검증 결과:

| 항목 | 결과 |
|---|---:|
| 전체 테스트 | 21개 |
| 통과 테스트 | 21개 |
| 실패 테스트 | 0개 |
| 통과율 | 100% |

세부 구성은 Google Test 기반 BMI 단위 테스트 20개와 Approval/Golden Master 테스트 1개입니다.

### 3.2 gcov/lcov 커버리지 검증

Windows/MSVC 빌드는 gcov/lcov와 직접 호환되지 않으므로 MinGW GCC 기반의 별도 `build-coverage` 디렉터리를 구성했습니다.

사용한 핵심 명령은 다음과 같습니다.

```powershell
cmake -S . -B build-coverage -G "MinGW Makefiles" `
  -DCMAKE_BUILD_TYPE=Debug `
  -DCMAKE_CXX_COMPILER="C:\mingw64\winlibs-x86_64-posix-seh-gcc-15.2.0-mingw-w64ucrt-14.0.0-r7\mingw64\bin\g++.exe" `
  -DCMAKE_CXX_FLAGS="--coverage -O0 -g" `
  -DCMAKE_EXE_LINKER_FLAGS="--coverage"

cmake --build build-coverage
ctest --test-dir build-coverage --output-on-failure
lcov --capture --directory build-coverage --output-file build-coverage\coverage.info
lcov --extract build-coverage\coverage.info "*/src/main/cpp/*" --output-file build-coverage\coverage.product.info
lcov --list build-coverage\coverage.product.info
```

제품 코드 기준 커버리지 결과:

| 대상 | 목표 | 실제 | 판단 |
|---|---:|---:|---|
| 전체 제품 코드 라인 | 90% 이상 | 91.5% (140/153) | 달성 |
| `SHealth.cpp` 라인 | 95% 이상 | 90.4% (122/135) | 미달 |
| `StandardBmiClassifier.cpp` 라인 | 100% | 100% (6/6) | 달성 |
| 제품 코드 함수 | 별도 목표 없음 | 100% (20/20) | 양호 |
| 분기 커버리지 | 85% 이상 | no data found | 측정 옵션 개선 필요 |

## 4. 최종 보고서 작성 내용

`docs/qa_final_report.md`에는 다음 내용을 정리했습니다.

### 4.1 테스트 완료율과 커버리지

실행 테스트 기준으로는 21/21 Green이며, 제품 코드 전체 라인 커버리지는 91.5%로 목표 90%를 달성했습니다.

반면 테스트 계획서의 P0/P1/P2 시나리오 기준 자동화 완료율은 전체 47.8%로 평가했습니다. 실행 테스트가 모두 통과하더라도, 계획 대비 남은 시나리오가 존재함을 분리해 명시했습니다.

### 4.2 결함 패턴 분석

`docs/defect_list.md` 기준 등록 결함 3건을 Severity와 ItemType으로 분석했습니다.

- Major / TestFailure: placeholder 실패 테스트 1건
- Info / DefectCandidate: 재현 불가 결함 후보 1건
- Info / Tooling: PowerShell 명령 호환성 문제 1건

또한 요구사항 분석과 코드 품질 분석에서 도출된 결함 후보를 `Functional`, `DataValidation`, `TestFailure`, `Tooling`, `Documentation` 패턴으로 분류했습니다.

가장 큰 반복 패턴은 `DataValidation`으로 판단했습니다. 컬럼 부족, 비숫자 값, 음수 체중, 키 0, 빈 줄, 전체 체중 0 같은 입력 검증 결함이 계산 오류와 런타임 위험으로 이어질 수 있기 때문입니다.

### 4.3 9단계 QA 활동 평가

9단계 활동을 다음과 같이 평가했습니다.

| 단계 | 평가 요약 |
|---:|---|
| 1 | 프로젝트 규칙/작업 기준 정리로 산출물 일관성 확보 |
| 2 | 요구사항 분석에서 핵심 경계값과 입력 리스크 조기 발견 |
| 3 | 코드 품질 분석으로 구조적 결함을 리팩토링 우선순위로 전환 |
| 4 | Modern C++ 리팩토링으로 안전성과 변경 용이성 개선 |
| 5 | 체중 누락 보정 규칙을 테스트로 고정 |
| 6 | 테스트 계획으로 목표 커버리지와 P0/P1/P2 기준 명확화 |
| 7 | Google Test 구현으로 placeholder 실패 테스트 제거 |
| 8 | 결함 분석/디버깅으로 재현 절차와 심각도 기준 정리 |
| 9 | Golden Master로 콘솔 출력 회귀 검증 확보 |

가장 효과적인 단계는 요구사항 분석, 코드 품질 분석, 테스트 계획, Golden Master로 정리했습니다. 개선이 필요한 단계는 테스트 구현 범위 확장과 커버리지 측정 자동화입니다.

### 4.4 Best Practice 5가지

다음 레거시 프로젝트를 위한 Best Practice로 5가지를 제안했습니다.

1. 요구사항을 테스트 가능한 경계값으로 먼저 변환한다.
2. 레거시 코드는 Golden Master를 먼저 만든 뒤 내부를 변경한다.
3. 결함은 Severity와 ItemType을 분리해 관리한다.
4. 커버리지 목표는 CMake/CI에서 실행 가능한 옵션으로 고정한다.
5. AI 산출물은 계획 대비 체크리스트로 검증한다.

### 4.5 Cursor AI 활용 효과

정량 효과는 다음과 같이 요약했습니다.

| 항목 | Before | After |
|---|---:|---:|
| 의미 있는 자동화 테스트 | 0개 또는 placeholder 실패 테스트 | 21개 Green |
| CTest 통과율 | 실패 상태 | 100% |
| 제품 코드 라인 커버리지 | 미측정 | 91.5% |
| `StandardBmiClassifier.cpp` 커버리지 | 미측정 | 100% |
| 등록 결함 해결 | placeholder 실패 1건 | Resolved |

정성 효과로는 결함 조기 발견, 테스트 가능 구조로의 리팩토링, 문서-테스트-결함 추적성 향상을 정리했습니다. 한계로는 AI가 생성한 테스트가 핵심 경계값에는 강했지만 테스트 계획 전체 자동화까지 완결하지 못했다는 점을 명시했습니다.

## 7. 후속 권장 사항

1. `SHealth.cpp` 커버리지를 95% 이상으로 올리기 위해 파일 없음, malformed row, 비숫자 입력, 빈 파일, 헤더만 있는 파일, unknown type, 반복 호출 테스트를 추가합니다.
2. lcov branch coverage 수집 옵션을 활성화해 분기 커버리지 85% 목표를 실제 측정합니다.
3. `CMakeLists.txt`에 `ENABLE_COVERAGE` 옵션을 추가해 커버리지 빌드 절차를 표준화합니다.
4. `docs/defect_list.md`의 ItemType 값을 `docs/defect_report.md`의 taxonomy와 맞춥니다.
5. 음수 나이, 헤더 없는 CSV, 표준 CSV quoting, 키 누락 보정 정책을 요구사항 문서에서 확정합니다.
