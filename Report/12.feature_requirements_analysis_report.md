## 4. 현재 구현 관찰 결과

현재 `SHealth`는 `calculateBmi()`를 중심으로 파일 로딩, 파싱, 체중 보정, BMI 계산, 연령대별 비율 산출을 수행합니다.

이미 일부 책임은 함수 단위로 분리되어 있습니다.

- `loadRecords()`: CSV 파일 로딩
- `parseRecord()`: CSV 행 파싱 및 기본 유효성 검사
- `imputeMissingWeights()`: 체중 `0` 평균 보정
- `calculateBmiValues()`: BMI 계산
- `aggregateRatios()`, `calculateRatioSet()`: 연령대별 BMI 비율 집계
- `BmiClassifier`, `StandardBmiClassifier`: BMI 범주 분류 책임 분리

다만 정상 BMI 사용자 목록 조회와 전체 BMI 분포 계산을 추가하려면 현재 private nested struct인 `HealthRecord`의 역할을 확장하거나 독립 모델로 분리해야 합니다. 특히 현재 구현은 CSV의 사용자 ID를 저장하지 않으므로, 정상 범위 사용자 목록 조회 기능을 위해서는 ID 보존이 선행되어야 합니다.

## 5. SRP 기반 개선 방향

| 책임 | 개선 후보 | 기대 효과 |
|---|---|---|
| 데이터 모델 | `HealthRecord`를 독립 모델로 분리하고 `id`, `age`, `weight`, `height`, `bmi` 보존 | 조회 API와 테스트에서 사용자 단위 결과 검증 가능 |
| CSV 처리 | `CsvHealthRecordReader` 분리 | 파일 open 실패, 빈 줄, 컬럼 부족, 비숫자 입력 정책 집중 관리 |
| 누락값 보정 | `MissingValueImputer` 분리 | 체중 0, 키 0 보정 정책을 동일한 구조로 관리 |
| BMI 계산 | `BmiCalculator` 또는 순수 함수 분리 | cm to m 변환과 BMI 수식 단위 테스트 용이 |
| BMI 통계 | `BmiStatisticsService` 분리 | 연령대별 분포와 전체 분포를 같은 기준으로 확장 가능 |
| BMI 분류 | 기존 `BmiClassifier` 유지 | 경계값 정책을 독립적으로 테스트 가능 |

## 6. 기능별 구현 요구사항

### 6.1 특정 연령대 BMI 분포 비율

특정 연령대의 유효 BMI 레코드를 저체중, 정상, 과체중, 비만으로 집계하고 `범주 건수 / 연령대 유효 건수 * 100`으로 계산합니다.

기존 `getBmiRatio(ageClass, type)`을 유지할 수 있지만, 장기적으로는 `type` 매직 넘버 대신 `BmiCategory` 기반 API를 제공하는 것이 더 안전합니다.

### 6.2 `height == 0` 평균치 보정

키가 `0`인 경우 누락값으로 간주하고, 같은 10년 단위 연령대의 `height > 0` 표본 평균으로 보정합니다.

체중 0 보정과 동일하게 평균 계산 시 보정 대상 값은 제외해야 하며, 같은 연령대에 유효 키가 하나도 없으면 해당 레코드는 BMI 통계에서 제외하는 정책이 필요합니다.

### 6.3 BMI 정상 범위 사용자 목록 조회

BMI 정상 범위는 `18.5 < bmi < 23.0`입니다. 따라서 BMI가 정확히 `18.5`인 사용자는 저체중, 정확히 `23.0`인 사용자는 과체중으로 분류되어 정상 목록에서 제외되어야 합니다.

반환 목록은 사용자 ID를 포함해야 하므로 CSV의 첫 번째 컬럼인 `id`를 파싱 단계에서 보존해야 합니다. 기본 정렬 정책은 입력 순서 유지로 정의했습니다.

### 6.4 전체 사용자 대비 BMI 범주 비율

연령대별 통계와 별도로 전체 유효 BMI 레코드를 분모로 사용해 각 BMI 범주 비율을 계산합니다.

이 기능은 기존 연령대별 비율과 분모가 다를 수 있으므로 `getOverallBmiDistribution()`처럼 API 이름에서 전체 기준임을 명확히 드러내야 합니다.

## 7. Google Test 시나리오 정리

`docs/feature_requirements_analysis.md`에는 기능개선 검증을 위한 `TC-FR-001`부터 `TC-FR-020`까지의 테스트 시나리오를 정의했습니다.

주요 테스트 그룹은 다음과 같습니다.

1. 정상 CSV 로딩과 기존 연령대별 BMI 비율 계산 검증
2. BMI `18.5`, `23.0`, `25.0` 경계값 분류 검증
3. 나이 `20`, `29`, `30`, `39`, `70`, `79`, `80` 경계값 검증
4. 체중 `0`과 키 `0`의 연령대 평균 보정 검증
5. 음수 체중, 음수 키, 컬럼 부족, 비숫자 입력 등 예외 데이터 검증
6. 정상 BMI 사용자 목록 조회 검증
7. 전체 사용자 대비 BMI 범주 비율 검증
8. 반복 호출과 파일 open 실패 시 상태 초기화 검증

## 8. 구현 우선순위

기능 구현은 다음 순서가 적절합니다.

1. `HealthRecord`에 사용자 ID를 포함하고 데이터 모델을 정리합니다.
2. 기존 체중 보정, BMI 계산, 연령대별 비율 테스트를 Green 상태로 고정합니다.
3. `height == 0` 평균 키 보정 로직과 테스트를 추가합니다.
4. 정상 BMI 사용자 목록 조회 API와 테스트를 추가합니다.
5. 전체 BMI 범주 비율 API와 테스트를 추가합니다.
6. 마지막으로 `SHealth` 내부 책임을 reader, imputer, statistics service 등으로 점진 분리합니다.

## 9. 검증 결과

이번 작업은 문서 작성 작업이므로 C++ 빌드나 Google Test 실행은 수행하지 않았습니다.

작성한 문서에 대해서는 linter 확인을 수행했으며, `docs/feature_requirements_analysis.md`에서 오류는 발견되지 않았습니다.

## 10. 결론

이번 분석의 핵심은 README의 기능개선 항목을 단순 TODO가 아니라 구현 가능한 C++ 책임 단위와 QA 검증 가능한 테스트 항목으로 바꾼 것입니다.

특히 `height == 0` 보정, 정상 BMI 사용자 목록 조회, 전체 BMI 범주 비율은 기존 `SHealth` API만으로는 표현이 부족하므로 데이터 모델 확장과 조회 API 추가가 필요합니다. 이후 구현 단계에서는 테스트를 먼저 고정한 뒤 작은 단위로 책임을 분리하는 방식이 가장 안전합니다.
