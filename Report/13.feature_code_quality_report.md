# SHEALTH BMI 코드 품질 분석 보고서

## 1. 작성 목적

본 문서는 `src/main/cpp/SHealth.cpp`에 정의된 메소드들을 시니어 C++ 아키텍트 및 모던 C++ 리뷰어 관점에서 분석한 결과를 보고서 형태로 요약합니다.

분석 기준은 SOLID 원칙, Code Smell, Magic Number, 조건문 복잡도, C++17 리팩토링 가능성입니다. 상세 분석 문서는 `docs/feature_code_quality_report.md`에 함께 작성했습니다.

## 3. 현재 구현 요약

현재 `SHealth`는 `calculateBmi()`를 중심으로 CSV 파일 로딩, 행 파싱, 체중 결측치 보정, BMI 계산, 연령대별 BMI 범주 비율 산출을 수행합니다.

이미 일부 개선은 적용되어 있습니다.

- `BmiClassifier`와 `StandardBmiClassifier`로 BMI 분류 책임을 분리했습니다.
- CSV delimiter, 연령대 범위, BMI type code, 컬럼 인덱스 등 주요 숫자는 익명 namespace 상수로 관리하고 있습니다.
- `calculateBmi()` 내부 흐름은 `loadRecords()`, `imputeMissingWeights()`, `calculateBmiValues()`, `aggregateRatios()`로 함수 분리되어 있습니다.

하지만 `SHealth` 클래스가 입력, 보정, 계산, 집계, 조회 책임을 모두 소유하고 있어 기능이 늘어날수록 SRP/OCP 위반이 커질 가능성이 높습니다.

## 4. 주요 문제점 분석

|---|---|---|---|---:|
| `SHealth`가 CSV 로딩, 파싱, 보정, BMI 계산, 통계 집계, 조회 API까지 담당한다. | SRP 위반, God Class 경향 | 기능 추가 시 `SHealth.cpp`가 계속 비대해지고 테스트 단위가 커진다. | `CsvHealthRecordReader`, `MissingValueImputer`, `BmiCalculator`, `BmiStatisticsService`로 책임 분리 | 1 |
| `calculateBmi()`가 전체 처리 파이프라인을 직접 조율한다. | SRP 위반, Long Method 가능성, Temporal Coupling | height 보정, 정상 BMI 사용자 조회, 전체 비율 계산이 추가되면 메소드가 빠르게 복잡해진다. | `load -> impute -> calculate -> aggregate` 단계를 별도 서비스 또는 순수 함수 조합으로 분리 | 1 |

| `parseRecord()`가 CSV의 ID 컬럼을 보존하지 않는다. | 요구사항 누락 위험, Primitive Obsession | 정상 BMI 사용자 목록 조회 시 사용자 식별자를 반환할 수 없다. | `HealthRecord`를 독립 모델로 분리하고 `id`, `age`, `weight`, `height`, `bmi` 명시 | 2 |

## 5. C++17 개선 방향

### 5.1 전략 패턴

`BmiClassifier`는 이미 전략 패턴으로 분리되어 있으므로 같은 방향을 변경 가능성이 높은 정책에 확장하는 것이 좋습니다.

| 전략 후보 | 역할 | 기대 효과 |
|---|---|---|
| `HealthRecordReader` | CSV 또는 다른 입력 소스에서 레코드 로딩 | 입력 형식 변경 시 `SHealth` 수정 최소화 |
| `MissingValuePolicy` | 체중/키 결측치 보정 정책 | 평균 보정, 스킵, 기본값 대입 정책을 독립 테스트 |
| `AgeClassPolicy` | 연령대 산정과 지원 범위 판단 | 20~70대, 10년 단위 정책을 한 곳에서 관리 |
| `BmiStatisticsService` | 연령대별/전체 BMI 비율 집계 | 신규 통계 API를 같은 구조로 확장 |

### 5.2 테이블 기반 매핑

`categoryFromType()`의 `switch`와 `SHealthBMI.cpp`의 반복 출력은 테이블 기반으로 개선할 수 있습니다.

```cpp
constexpr std::array<std::pair<int, BmiCategory>, 4> BmiTypeMappings{{
    {100, BmiCategory::Underweight},
    {200, BmiCategory::Normal},
    {300, BmiCategory::Overweight},
    {400, BmiCategory::Obesity},
}};
```

이 구조를 사용하면 외부 type code와 내부 `BmiCategory` 관계를 한 곳에서 관리할 수 있고, 조회 API와 출력 코드가 같은 매핑을 공유할 수 있습니다.

### 5.3 `std::variant` 적용 가능성

현재 리팩토링의 1차 우선순위는 `std::variant`가 아니라 책임 분리와 명시적 도메인 타입 도입입니다. 다만 다음 경우에는 `std::variant`가 유용할 수 있습니다.

| 적용 후보 | 예시 | 판단 |
|---|---|---|
| 파싱 결과 | `std::variant<HealthRecord, ParseError>` | 실패 사유를 보존하면서 예외 없는 파싱 흐름을 만들 때 적합 |
| 입력 소스 | `std::variant<CsvFileInput, InMemoryInput>` | 입력 타입이 소수로 고정되어 있고 런타임 다형성을 피하고 싶을 때 적합 |
| 통계 요청 | `std::variant<AgeClassDistributionRequest, OverallDistributionRequest, NormalUsersRequest>` | 조회 API가 늘어나고 요청별 처리를 visitor로 분리하고 싶을 때 고려 |

## 6. 리팩토링 우선순위

| 우선순위 | 작업 | 이유 |
|---:|---|---|
| 1 | CSV 읽기/파싱과 `SHealth` 도메인 흐름 분리 | SRP 위반의 중심이며 테스트 단위 개선 효과가 가장 크다. |
| 2 | 결측치 보정과 연령대 그룹핑 정책 분리 | height 보정과 전체 통계 기능 추가 시 중복과 조건 누락을 줄인다. |
| 3 | BMI type int API를 enum/table 기반으로 격리 | OCP 위반과 magic number 노출을 줄이고 호출 안정성을 높인다. |
| 4 | 통계 집계를 `BmiStatisticsService`로 분리 | 연령대별 비율, 전체 비율, 정상 BMI 목록 조회를 같은 구조로 확장할 수 있다. |
| 5 | `SHealthBMI.cpp` 출력 반복 제거 | 실행 예시는 깔끔해지지만 핵심 도메인 안정성에 비해 영향이 작다. |

## 7. 결론

현재 구현은 `BmiClassifier` 분리와 상수화가 일부 적용되어 있어 리팩토링 기반은 마련되어 있습니다. 그러나 `SHealth`가 여전히 여러 책임을 동시에 수행하므로 신규 기능을 추가할수록 변경 범위가 커질 가능성이 높습니다.

가장 먼저 CSV 파싱과 결측치/연령대 정책을 분리하고, 이후 BMI type mapping과 통계 서비스를 정리하는 순서가 적절합니다. 이 순서로 진행하면 기존 public 동작을 보존하면서도 SRP/OCP 위반과 중복 코드 위험을 단계적으로 줄일 수 있습니다.
