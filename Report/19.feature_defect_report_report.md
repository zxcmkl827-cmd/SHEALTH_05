# SHealth 기능 개선 결함 관리 문서 작성 보고서

## 1. 작성 목적

본 문서는 `SHealth` BMI 기능 개선 작업의 결함 관리 체계를 `docs/feature_defect_report.md`로 작성한 결과를 보고하기 위한 문서입니다.

주요 목적은 다음과 같습니다.

- 기능 개선 요구사항과 테스트 계획에 맞는 결함 분류 체계 수립
- 결함 등록 시 사용할 표준 보고서 템플릿 정의
- 테스트 통과율, 커버리지, 단계별 결함 발견율 등 품질 메트릭 수집 기준 정리
- C++ 프로젝트에 적합한 `gcov`/`lcov` 커버리지 수집 절차 문서화
- GitHub Issues 기반 결함 추적 워크플로우 제안

## 2. 입력 문서

결함 관리 문서는 다음 자료를 기준으로 작성했습니다.

| 문서 | 반영 내용 |
|---|---|
| `docs/feature_defect_list.md` | 현재 결함 목록, `Info / Tooling`, `Info / TestFailure` 항목, 전체 테스트 Green 상태 |
| `docs/feature_requirements_analysis.md` | 기능 개선 범위, BMI 계산/보정/조회/통계 API 요구사항 |
| `docs/feature_test_plan.md` | P0/P1/P2 테스트 우선순위, 경계값 목록, 커버리지 목표, `gcov`/`lcov` 전략 |

현재 기준으로 `src/test/cpp/SHealthBMITest.cpp`의 Google Test 40개와 Golden Master 1개는 모두 통과했으며, 재현 가능한 제품 기능 결함은 없는 상태를 전제로 정리했습니다.

## 3. 작성 산출물

신규 작성한 산출물은 다음과 같습니다.

| 산출물 | 설명 |
|---|---|
| `docs/feature_defect_report.md` | 기능 개선 결함 관리 기준 문서 |

문서에는 다음 항목을 포함했습니다.

- `Severity` 4단계 정의: `Critical`, `Major`, `Minor`, `Info`
- `ItemType` 5종 정의: `Functional`, `DataValidation`, `Boundary`, `TestFailure`, `Tooling`
- `Severity x ItemType` 결함 분류 매트릭스
- 결함 보고서 템플릿: 재현, 기대, 실제, 원인, 수정, 검증
- 테스트 통과율과 회귀 테스트 통과율 수집 계획
- `gcov`/`lcov` 기반 커버리지 측정 절차
- 단계별 결함 발견율 산식과 보조 품질 지표
- GitHub Issues 라벨, 상태, PR 연동 방식

## 4. 결함 분류 체계 요약

결함 심각도는 출시 영향도와 품질 게이트 차단 여부를 기준으로 나누었습니다.

| Severity | 기준 |
|---|---|
| Critical | 핵심 BMI 계산, 데이터 보정, 통계 산출이 중단되거나 출시를 막는 결함 |
| Major | 주요 요구사항이 일부 실패하거나 특정 조건에서 잘못된 결과가 발생하는 결함 |
| Minor | 영향 범위가 제한적이고 우회 가능한 결함 |
| Info | 제품 결함은 아니지만 테스트 환경, 문서, 도구 개선이 필요한 항목 |

결함 유형은 기능 개선 요구사항과 테스트 실패 양상을 기준으로 5종으로 정리했습니다.

| ItemType | 기준 |
|---|---|
| Functional | 공개 API 동작이 요구사항과 다른 결함 |
| DataValidation | CSV 파싱, 잘못된 값, 누락값 보정 정책 처리 결함 |
| Boundary | BMI, 나이, 분모 0 등 경계값 처리 결함 |
| TestFailure | Google Test, Golden Master, flaky test 등 자동화 테스트 실패 |
| Tooling | 빌드, 테스트 실행, 커버리지, CI 환경 문제 |

이 분류를 조합한 매트릭스를 통해 동일한 `ItemType`이라도 출시 영향도에 따라 `Critical`부터 `Info`까지 구분할 수 있도록 했습니다.

## 5. 결함 보고 템플릿 요약

결함 등록 템플릿은 QA 리드 관점에서 재현성과 검증 가능성을 중심으로 구성했습니다.

필수 항목은 다음과 같습니다.

| 항목 | 목적 |
|---|---|
| 재현 절차 | 동일 환경에서 결함을 다시 발생시킬 수 있도록 입력 CSV, 테스트명, 실행 명령을 기록 |
| 기대 결과 | 요구사항 또는 테스트 계획 기준의 정상 동작 기록 |
| 실제 결과 | 실패 메시지, 반환값, diff, 로그 등 관찰된 현상 기록 |
| 원인 분석 | 코드 위치, 입력 조건, 정책 누락, 환경 차이 구분 |
| 수정 내용 | 최소 수정 범위와 영향받는 API 또는 테스트 기록 |
| 검증 결과 | 실행 명령, 테스트 결과, 커버리지 영향, 회귀 확인 기록 |

이 템플릿은 `docs/feature_defect_list.md`의 항목 확장 또는 GitHub Issue 본문으로 재사용할 수 있습니다.

## 6. 품질 메트릭 수집 계획 요약

품질 메트릭은 테스트 성공 여부, 커버리지 달성 여부, 결함 발견 단계의 세 축으로 정의했습니다.

### 6.1 테스트 통과율

| 메트릭 | 목표 |
|---|---:|
| 전체 테스트 통과율 | 100% |
| P0 테스트 통과율 | 100% |
| 회귀 테스트 통과율 | 100% |
| Golden Master 통과 여부 | Pass |

Windows PowerShell 환경에서는 다음 명령을 권장 절차로 문서화했습니다.

```powershell
cmake --build build; if ($LASTEXITCODE -eq 0) { ctest --test-dir build --output-on-failure }
```

기존 결함 목록의 `FDEF-001`에서 확인된 것처럼, PowerShell 환경에서는 `&&` 문법이 호환되지 않을 수 있으므로 `$LASTEXITCODE` 기반 절차를 기준으로 삼았습니다.

### 6.2 커버리지

커버리지 목표는 `docs/feature_test_plan.md`의 기준을 그대로 연결했습니다.

| 대상 | 목표 |
|---|---:|
| `src/main/cpp/SHealth.cpp` 라인 커버리지 | 90% 이상 |
| `src/main/cpp/SHealth.cpp` 브랜치 커버리지 | 85% 이상 |
| `StandardBmiClassifier.cpp` 라인 커버리지 | 95% 이상 |
| 신규 기능 코드 라인 커버리지 | 90% 이상 |
| 전체 `src/main/cpp` 라인 커버리지 | 90% 이상 |

측정 도구는 C++ 프로젝트 특성에 맞춰 GCC 또는 MinGW 기반 `gcov`/`lcov`를 사용하도록 정리했습니다. Windows 로컬에서 `lcov` 사용이 어려운 경우에는 WSL 또는 Linux CI 결과를 공식 커버리지 기준으로 관리하도록 명시했습니다.

### 6.3 단계별 결함 발견율

결함 발견 단계는 다음 6단계로 정의했습니다.

| 단계 | 목적 |
|---|---|
| 요구사항 분석 | 정책 불명확, API 반환 범위 모호성 조기 발견 |
| 테스트 설계 | 누락된 경계값, 예외 케이스, 회귀 시나리오 발견 |
| 구현 및 단위 테스트 | Google Test 실행 중 기능 결함 발견 |
| 통합 검증 | Golden Master, 전체 CTest, 반복 호출 결함 발견 |
| 커버리지 및 CI | 미검증 분기, 환경 문제, 품질 게이트 실패 발견 |
| 릴리스 전 리뷰 | 잔여 위험, 문서 누락, 재현 불가 결함 정리 |

산식은 다음과 같이 정의했습니다.

```text
단계별 결함 발견율 = 해당 단계에서 최초 발견된 결함 수 / 전체 발견 결함 수 * 100
```

보조 지표로는 Severity 분포, ItemType 분포, 결함 제거율, 재오픈율, 결함 수정 리드타임, 커버리지 미달 항목 수를 제안했습니다.

## 7. GitHub Issues 연동 워크플로우

선택 항목으로 GitHub Issues 기반 결함 추적 흐름을 정리했습니다.

권장 흐름은 다음과 같습니다.

1. 결함 발견 시 표준 템플릿으로 Issue 생성
2. Severity 라벨과 ItemType 라벨 지정
3. 관련 테스트 ID와 실패 로그 또는 Golden Master diff 첨부
4. 수정 PR 본문에 `Fixes #이슈번호` 기록
5. PR 검증 항목에 CTest, 커버리지, 회귀 테스트 추가 여부 포함
6. QA 확인 후 `Verified` 상태로 전환하고 종료

권장 상태는 `Open`, `In Progress`, `Fixed`, `Verified`, `Closed`, `Reopened`입니다. 코드 수정 완료 상태인 `Fixed`와 QA 검증 완료 상태인 `Verified`를 구분해 결함 종료 기준을 명확히 했습니다.
