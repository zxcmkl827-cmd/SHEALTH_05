## 3. 수행한 검증

### 3.1 기본 빌드 및 CTest

다음 PowerShell 호환 명령으로 기본 빌드와 테스트를 실행했습니다.

```powershell
cmake --build build; if ($LASTEXITCODE -eq 0) { ctest --test-dir build --output-on-failure }
```

검증 결과:

| 항목 | 결과 |
|---|---:|
| Google Test 기능 테스트 | 40개 통과 |
| Golden Master 테스트 | 2개 통과 |
| 전체 CTest | 42개 통과 |
| 실패 테스트 | 0개 |
| 전체 테스트 통과율 | 100% |

### 3.2 커버리지 빌드 및 gcov/lcov

기존 `.gcda` 체크섬 충돌 가능성을 제거하기 위해 `build-coverage`를 clean 후 다시 빌드하고 테스트를 실행했습니다.

```powershell
cmake --build build-coverage --target clean
cmake --build build-coverage
ctest --test-dir build-coverage --output-on-failure
```

이후 다음 명령으로 제품 코드 기준 커버리지를 산출했습니다.

```powershell
lcov --capture --directory build-coverage --rc lcov_branch_coverage=1 --output-file build-coverage\coverage.info
lcov --extract build-coverage\coverage.info "*/src/main/cpp/*" --rc lcov_branch_coverage=1 --output-file build-coverage\coverage.product.info
lcov --list build-coverage\coverage.product.info --rc lcov_branch_coverage=1
```

커버리지 결과:

| 대상 | 목표 | 실제 | 판단 |
|---|---:|---:|---|
| 전체 `src/main/cpp` 라인 커버리지 | 90% 이상 | 96.1% (196/204) | 달성 |
| `SHealth.cpp` 라인 커버리지 | 90% 이상 | 95.7% (178/186) | 달성 |
| `StandardBmiClassifier.cpp` 라인 커버리지 | 95% 이상 | 100.0% (6/6) | 달성 |
| 제품 코드 함수 커버리지 | 별도 목표 없음 | 100.0% (25/25) | 양호 |
| 전체 제품 코드 브랜치 커버리지 | 85% 이상 | 66.9% (176/263) | 미달 |
| `SHealth.cpp` 브랜치 커버리지 | 85% 이상 | 71.7% (137/191) | 미달 |

### 4.1 종합 결론

SHEALTH C++ 기능 개선 QA 활동은 테스트 가능한 요구사항 분해, Google Test 자동화, Golden Master 회귀 검증, 커버리지 측정을 통해 기능 릴리스 후보로는 Green 상태라고 판단했습니다.

다만 브랜치 커버리지가 목표 85%에 미달하므로, QA 리드 관점에서는 조건부 승인 기준을 제시했습니다.

### 4.3 결함 패턴 분석

`docs/feature_defect_list.md` 기준 등록 결함 2건은 모두 정보성 항목입니다.

| 결함 | Severity | ItemType | 의미 |
|---|---|---|---|
| FDEF-001 | Info | Tooling | PowerShell `&&` 구문 호환성 이슈 |
| FDEF-002 | Info | TestFailure | 실패 로그 부재와 현재 Green 상태 기록 |

현재 재현 가능한 제품 결함은 없지만, 잠재 결함 패턴은 `DataValidation`, `Boundary`, `Functional`, `TestFailure`, `Tooling`으로 정리했습니다. 특히 CSV 파싱 실패, 누락값 보정, 분모 0, 나이/BMI 경계값을 핵심 QA 리스크로 보았습니다.

### 4.4 9단계 QA 활동 평가

9단계 활동은 다음 흐름으로 정리했습니다.

1. 요구사항 분석
2. 코드 품질 분석
3. 테스트 계획 수립
4. 테스트 fixture 설계
5. 기능 테스트 구현
6. 기능 구현 및 보강
7. Golden Master 회귀 검증
8. 결함 분석 및 관리
9. 커버리지 측정 및 품질 게이트

가장 효과적이었던 단계는 요구사항 분석, 테스트 계획, 기능 테스트 구현, Golden Master 회귀 검증으로 평가했습니다. 개선 필요 단계는 구조 분리, 결함 후보 backlog화, 브랜치 커버리지 자동화입니다.

### 4.5 Best Practice 5가지

다음 레거시 프로젝트를 위한 Best Practice는 다음 5가지로 정리했습니다.

1. 요구사항을 먼저 테스트 가능한 경계값과 실패 조건으로 바꾼다.
2. Golden Master로 기존 외부 동작을 잠근 뒤 내부를 바꾼다.
3. Severity와 ItemType을 분리해 결함을 분류한다.
4. 라인 커버리지보다 브랜치 커버리지 미달 경로를 먼저 리뷰한다.
5. AI가 만든 산출물은 계획 대비 추적표로 검증한다.

## 7. 후속 권장 사항

1. 브랜치 커버리지 85% 목표 달성을 위해 negative test를 추가합니다.
2. malformed CSV, 빈 파일, 헤더 전용 파일, 비숫자 필드 테스트를 보강합니다.
3. `SHealth` 내부의 CSV 파싱, 결측치 보정, 통계 집계를 별도 클래스로 분리합니다.
