# SHEALTH C++ 기능 개선 QA 최종 보고서

작성일: 2026-05-20  
관점: QA 리드 엔지니어  
대상: `SHealth` BMI 기능 개선, 테스트 자동화, 결함 관리, 커버리지 검증

## 1. 종합 결론

SHEALTH C++ 프로젝트의 BMI 기능 개선 QA 활동은 기능 요구사항을 테스트 가능한 경계값과 예외 조건으로 분해하고, Google Test와 Golden Master를 함께 사용해 회귀 안정성을 확보한 점이 가장 큰 성과다. 최신 검증 기준으로 CTest 전체 42개 테스트가 모두 통과했으며, `gcov`/`lcov` 기준 제품 코드 라인 커버리지는 96.1%로 목표 90%를 달성했다.

다만 브랜치 커버리지는 66.9%로 목표 85%에 미달한다. 현재 테스트는 핵심 사용자 시나리오와 경계값에는 강하지만, `catch (...)`, 생성자 fallback, CSV quoting/빈 파일/헤더만 있는 파일, 일부 잘못된 타입 코드와 같은 방어 분기까지 완전히 검증하지는 못한다. 다음 단계의 품질 개선은 라인 커버리지 확대보다 분기 커버리지와 입력 파서의 음성 케이스 보강에 집중해야 한다.

## 2. 테스트 완료율 및 커버리지

### 2.1 테스트 실행 결과

검증 명령:

```powershell
cmake --build build; if ($LASTEXITCODE -eq 0) { ctest --test-dir build --output-on-failure }
cmake --build build-coverage --target clean; cmake --build build-coverage; ctest --test-dir build-coverage --output-on-failure
```

| 구분 | 결과 | 판단 |
|---|---:|---|
| Google Test 기능 테스트 | 40/40 통과 | 달성 |
| Golden Master 회귀 테스트 | 2/2 통과 | 달성 |
| 전체 CTest | 42/42 통과 | 달성 |
| 테스트 통과율 | 100% | 목표 100% 달성 |
| 실패 테스트 | 0개 | 출시 차단 결함 없음 |

### 2.2 테스트 계획 대비 완료율

`docs/feature_test_plan.md`의 P0/P1/P2 항목을 기준으로 자동화 상태를 평가했다.

| 우선순위 | 계획 항목 | 자동화 완료 | 완료율 | 비고 |
|---|---:|---:|---:|---|
| P0 | 7 | 7 | 100.0% | 핵심 기능, 보정, 조회, 전체 분포, 상태 초기화 검증 완료 |
| P1 | 10 | 9 | 90.0% | malformed CSV 세부 케이스 보강 필요 |
| P2 | 7 | 3 | 42.9% | 헤더 전용/빈 줄/전체 무효 BMI/연령대 API invalid type 등 보강 필요 |
| 합계 | 24 | 19 | 79.2% | 실행 테스트는 모두 Green이나 계획 전체 자동화는 추가 여지 있음 |

`docs/feature_requirements_analysis.md`의 20개 TC-FR 시나리오 관점에서는 주요 기능 요구사항은 모두 테스트로 다뤄졌다. 다만 `TC-FR-008`, `TC-FR-012`, `TC-FR-018`, `TC-FR-020`은 일부 조건만 검증되어 완전 자동화로 보려면 세부 테스트를 추가해야 한다.

### 2.3 gcov/lcov 커버리지

측정 명령:

```powershell
lcov --capture --directory build-coverage --rc lcov_branch_coverage=1 --output-file build-coverage\coverage.info
lcov --extract build-coverage\coverage.info "*/src/main/cpp/*" --rc lcov_branch_coverage=1 --output-file build-coverage\coverage.product.info
lcov --list build-coverage\coverage.product.info --rc lcov_branch_coverage=1
```

| 대상 | 목표 | 실제 | 판단 |
|---|---:|---:|---|
| 전체 `src/main/cpp` 라인 커버리지 | 90% 이상 | 96.1% (196/204) | 달성 |
| `SHealth.cpp` 라인 커버리지 | 90% 이상 | 95.7% (178/186) | 달성 |
| `StandardBmiClassifier.cpp` 라인 커버리지 | 95% 이상 | 100.0% (6/6) | 달성 |
| 제품 코드 함수 커버리지 | 별도 목표 없음 | 100.0% (25/25) | 양호 |
| 전체 제품 코드 브랜치 커버리지 | 85% 이상 | 66.9% (176/263) | 미달 |
| `SHealth.cpp` 브랜치 커버리지 | 85% 이상 | 71.7% (137/191) | 미달 |

브랜치 커버리지 미달은 출시 차단 수준의 기능 결함은 아니지만, QA 게이트 관점에서는 명확한 잔여 리스크다. 특히 CSV 파싱 실패, 파일 open 실패 이후 전체 조회 API, 생성자에 null classifier 전달, `getBmiRatio()`의 invalid type/unknown age class 조합, `SHealthBMI.cpp` 출력 분기 등 방어 경로 테스트가 더 필요하다.

## 3. 결함 패턴 분석

### 3.1 등록 결함 현황

`docs/feature_defect_list.md` 기준 현재 등록 항목은 2건이며, 모두 제품 기능 결함이 아닌 정보성 항목이다.

| ID | Severity | ItemType | 패턴 | 상태 |
|---|---|---|---|---|
| FDEF-001 | Info | Tooling | PowerShell `&&` 구문 호환성 문제 | 우회 절차 문서화 |
| FDEF-002 | Info | TestFailure | 실패 로그 부재 및 현재 Green 상태 기록 | 실제 실패 재현 시 재분석 |

Severity 분포는 `Info` 2건, `Critical/Major/Minor` 0건이다. ItemType 분포는 `Tooling` 1건, `TestFailure` 1건이며, 재현 가능한 `Functional`, `DataValidation`, `Boundary` 제품 결함은 없다.

### 3.2 잠재 결함 패턴

실제 결함은 재현되지 않았지만, 요구사항 분석과 코드 품질 분석에서 반복적으로 드러난 위험 패턴은 다음과 같다.

| 패턴 | 관찰 내용 | QA 판단 |
|---|---|---|
| DataValidation | CSV 컬럼 부족, 비숫자 값, 음수 값, `height == 0`, `weight == 0` 정책이 계산 정확도와 직접 연결됨 | 가장 중요한 방어 테스트 영역 |
| Boundary | BMI `18.5`, `23.0`, `25.0`, 나이 `19/20/79/80`, 분모 0이 요구사항 결과를 바꿈 | 현재 자동화가 강한 영역이나 일부 P2 보강 필요 |
| Functional | 정상 BMI 사용자 목록, 전체 분포, 연령대별 분포가 서로 다른 분모와 반환 정책을 가짐 | API별 기준 집합을 테스트명과 문서에 계속 명시해야 함 |
| TestFailure | Golden Master diff나 `EXPECT_EQ` 실패는 현재 없음 | 실패 로그 확보 시 재현 절차 중심으로 결함 등록 |
| Tooling | Windows PowerShell과 Unix 계열 명령 문법 차이, lcov branch 옵션 차이 | CI 스크립트 표준화 필요 |

결함 관리 관점에서 가장 의미 있는 성과는 제품 결함을 많이 찾은 것이 아니라, 결함이 발생하기 쉬운 입력 검증/경계값 영역을 테스트 설계 단계에서 먼저 고정했다는 점이다.

## 4. 9단계 QA 활동 평가

| 단계 | 활동 | 효과성 평가 | 개선 필요 사항 |
|---:|---|---|---|
| 1 | 요구사항 분석 | 기능을 `TC-FR-001`~`TC-FR-020`으로 분해해 테스트 추적성을 확보했다. | 전체 통계에 연령대 외 사용자를 포함하는 정책처럼 늦게 확정된 기준은 요구사항에 더 일찍 고정해야 한다. |
| 2 | 코드 품질 분석 | SRP 위반, CSV 파싱 결합, ID 미보존 같은 구조 리스크를 기능 결함 후보로 전환했다. | 분석 결과가 즉시 리팩토링 backlog로 연결되도록 우선순위와 담당 기준이 필요하다. |
| 3 | 테스트 계획 수립 | P0/P1/P2, 경계값, 예외 케이스, 커버리지 목표가 명확해졌다. | P2도 자동화 여부를 추적할 수 있게 체크박스 또는 상태 컬럼을 유지해야 한다. |
| 4 | 테스트 fixture 설계 | 임시 CSV 기반 fixture로 기능 테스트를 빠르게 확장했다. | malformed row, 빈 파일, 헤더 전용 파일을 생성하는 공통 헬퍼가 더 있으면 누락이 줄어든다. |
| 5 | 기능 테스트 구현 | 40개 Google Test가 BMI 분류, 보정, 조회, 전체 분포를 회귀 테스트로 고정했다. | 테스트 수는 충분하지만 일부 테스트가 public API 통합 검증에 집중되어 실패 원인 국소화가 어렵다. |
| 6 | 기능 구현 및 보강 | `height == 0` 보정, `BmiUser`, 정상 BMI 사용자, 전체 분포 API가 구현됐다. | `SHealth`가 여전히 파싱/보정/계산/집계를 모두 조율하므로 추가 기능 시 비대화 위험이 남아 있다. |
| 7 | Golden Master 회귀 검증 | CLI 출력 회귀를 2개 테스트로 보호해 기존 동작 변경 리스크를 낮췄다. | 승인 파일 갱신 기준과 의도된 출력 변경 승인 절차를 문서화해야 한다. |
| 8 | 결함 분석 및 관리 | Severity/ItemType 체계와 PowerShell 절차 이슈를 기록해 재현성을 높였다. | 실제 제품 결함이 없더라도 잠재 결함 후보를 backlog 항목으로 전환하는 흐름이 필요하다. |
| 9 | 커버리지 측정 및 품질 게이트 | 제품 코드 라인 96.1%, 함수 100%를 확인해 테스트 효과를 정량화했다. | 브랜치 66.9%는 목표 미달이므로 CI에서 branch coverage를 공식 게이트로 수집해야 한다. |

가장 효과적이었던 단계는 1단계 요구사항 분석, 3단계 테스트 계획, 5단계 기능 테스트 구현, 7단계 Golden Master다. 개선이 필요한 단계는 6단계 구조 분리, 8단계 결함 후보 backlog화, 9단계 브랜치 커버리지 자동화다.

## 5. 다음 레거시 프로젝트 Best Practice 5가지

1. 요구사항을 먼저 테스트 가능한 경계값과 실패 조건으로 바꾼다.  
   레거시 코드는 요구사항 문장이 모호할수록 결함 재현이 늦어진다. BMI `18.5/23.0/25.0`, 나이 `19/20/79/80`, 분모 0처럼 결과가 바뀌는 값을 먼저 표로 고정해야 한다.

2. Golden Master로 기존 외부 동작을 잠근 뒤 내부를 바꾼다.  
   콘솔 출력, 파일 포맷, 리포트 텍스트처럼 사용자가 보는 결과는 단위 테스트만으로 놓치기 쉽다. 리팩토링 전후 출력 diff를 승인 테스트로 관리하면 레거시 변경의 심리적 비용이 낮아진다.

3. Severity와 ItemType을 분리해 결함을 분류한다.  
   `Boundary` 결함도 출시 차단일 수 있고 정보성일 수 있다. 영향도와 유형을 분리하면 수정 우선순위, 재발 방지 테스트, 보고 대상이 명확해진다.

4. 라인 커버리지보다 브랜치 커버리지 미달 경로를 먼저 리뷰한다.  
   이번 프로젝트는 제품 코드 라인 96.1%를 달성했지만 브랜치는 66.9%에 그쳤다. 레거시 품질 리스크는 보통 정상 흐름보다 파서 실패, fallback, 오류 반환, 빈 데이터 경로에 남는다.

5. AI가 만든 산출물은 계획 대비 추적표로 검증한다.  
   테스트가 많아 보여도 P2나 malformed input이 빠질 수 있다. 요구사항 ID, 테스트 ID, 결함 ID, 커버리지 미달 라인을 연결해 사람이 최종 QA 판단을 내려야 한다.

## 6. Cursor AI 활용 효과

### 6.1 정량 요약

| 항목 | Before | After | 효과 |
|---|---:|---:|---|
| 기능 개선 Google Test | 기존 BMI 중심 테스트 | 40개 Green | 신규 요구사항 자동화 확대 |
| Golden Master 테스트 | 제한적 또는 미정착 | 2개 Green | CLI 출력 회귀 보호 |
| 전체 CTest | 기능 개선 전 기준 불명확 | 42/42 통과 | 품질 게이트 100% |
| 제품 코드 라인 커버리지 | 미측정 | 96.1% | 목표 90% 초과 |
| `SHealth.cpp` 라인 커버리지 | 미측정 | 95.7% | 목표 90% 초과 |
| 제품 코드 함수 커버리지 | 미측정 | 100.0% | 모든 함수 실행 |
| 제품 코드 브랜치 커버리지 | 미측정 | 66.9% | 측정은 가능해졌으나 목표 미달 |
| 재현 가능한 제품 결함 | 확인 필요 | 0건 | 출시 차단 결함 없음 |
| 정보성 결함/절차 이슈 | 미정리 | 2건 문서화 | 반복 실행 오류 예방 |

시간 단축 효과는 별도 작업 시간 로그가 없으므로 보수 추정으로만 해석해야 한다. 요구사항 분석, 테스트 계획, 결함 분류, 40개 단위 테스트와 2개 Golden Master 검증, 커버리지 보고까지 수작업으로 진행했다면 최소 수일 단위 작업이었을 가능성이 높다. Cursor AI를 사용하면서 같은 산출물을 짧은 반복 주기로 생성, 실행, 보정할 수 있었고 QA 리드는 최종 판단과 누락 검토에 시간을 더 쓸 수 있었다.

### 6.2 정성 요약

Cursor AI의 가장 큰 효과는 결함 조기 발견의 범위를 넓힌 것이다. 요구사항 분석 단계에서 ID 보존 누락, `height == 0` 보정 정책, 전체 분포와 연령대별 분포의 분모 차이를 먼저 드러냈고, 이를 테스트명과 public API 검증으로 연결했다.

두 번째 효과는 회귀 안정성이다. 기능 구현 중에도 기존 BMI 분류 경계값, 나이 경계값, 체중 보정, 정상 BMI 목록, 전체 분포, Golden Master 출력이 함께 검증되어 변경 영향이 빠르게 드러나는 구조가 되었다.

세 번째 효과는 문서-테스트-결함의 연결성이다. 요구사항 분석, 테스트 계획, 결함 보고서, 결함 목록, 커버리지 측정이 서로 같은 용어와 기준을 사용하면서 QA 산출물이 단순 문서가 아니라 실행 가능한 품질 기준으로 작동했다.

한계도 분명하다. AI는 정상 흐름과 대표 경계값 테스트를 빠르게 확장했지만, P2 입력 다양성이나 브랜치 커버리지 미달 경로까지 자동으로 완결하지는 못했다. 따라서 다음 레거시 프로젝트에서는 AI 생성 테스트 이후 반드시 커버리지 미달 라인/분기 기반의 사람 주도 보강 단계가 필요하다.
